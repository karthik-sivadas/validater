# Phase 6.1: Step Details, Screenshots, and Browser Experience - Research

**Researched:** 2026-03-07
**Domain:** Data pipeline enrichment, screenshot persistence, CDP streaming quality, frontend display
**Confidence:** HIGH

## Summary

This phase addresses three gaps left by Phase 6: (1) step events lack action type and description metadata, (2) screenshots are captured but stripped before Temporal returns them (so they never reach the database), and (3) CDP screencast quality uses conservative settings that produce flickery low-quality frames.

The standard approach for each gap is straightforward and uses existing infrastructure:

1. **Step metadata** -- The `TestStep` type already has `action` (TestStepAction) and `description` (string) fields. The `StepResult` type does not carry these. The fix is to add `action` and `description` to `StepResult`, propagate through the `StepEvent` type for live streaming, add columns to `test_run_steps` DB schema, and update the UI components.

2. **Screenshot persistence** -- Screenshots are captured in `step-runner.ts` but stripped in `execute-steps.activity.ts:109` before returning through Temporal (to avoid the 2MB gRPC payload limit). The fix is to persist screenshots directly to the database from within the `executeStepsActivity` function, before stripping them from the Temporal return payload. This uses the established factory DI pattern to inject the DB client into the execute-steps activity.

3. **Streaming quality** -- Current CDP settings are `quality: 60, everyNthFrame: 2`. Increasing quality to 80 and setting `everyNthFrame: 1` improves smoothness. Client-side flicker is addressed by using a canvas element with double-buffering instead of updating an `<img>` src attribute.

**Primary recommendation:** Enrich the step data pipeline end-to-end (core types -> worker activity -> Redis pub/sub -> DB schema -> frontend), persist screenshots directly from the activity before Temporal strips them, and improve streaming with better CDP params and canvas rendering.

## Architecture Patterns

### Data Flow: Step Metadata Pipeline

The step metadata needs to flow through 5 layers:

```
1. Core Types     StepResult gains `action` and `description` fields
       |
2. Step Runner    executeStep() copies action/description from TestStep to StepResult
       |
3. Worker Activity   onStepComplete callback includes action/description in StepEvent
       |                  (also published to Redis for live streaming)
       |
4. DB Schema      test_run_steps gains `action` and `description` columns
       |              (persist-results.activity.ts writes them)
       |
5. Frontend       LiveViewer shows action badges, Results page shows action+description
```

### Pattern 1: Enrich StepResult at Source
**What:** Add `action` and `description` to StepResult in `step-runner.ts` where the TestStep is already in scope, rather than trying to join this data later.
**When to use:** Whenever downstream consumers need data that's available at the execution point.
**Example:**
```typescript
// Source: packages/core/src/execution/step-runner.ts (modified)
return {
  stepId: step.id,
  stepOrder: step.order,
  action: step.action,           // NEW: propagate action type
  description: step.description, // NEW: propagate description
  status,
  error,
  screenshotBase64,
  durationMs: Date.now() - startTime,
};
```

### Pattern 2: Persist Before Temporal Return (Screenshot Side-Channel)
**What:** Write screenshots directly to the database inside the `executeStepsActivity` before stripping them from the Temporal return payload. This avoids the 2MB gRPC payload limit while still persisting data.
**When to use:** When activity produces large data that must be persisted but cannot traverse Temporal's payload limits.
**Example:**
```typescript
// Source: packages/worker/src/activities/execute-steps.activity.ts (modified)
// After executeSteps() returns stepResults with screenshots:

// Persist screenshots directly to DB (side-channel, bypasses Temporal payload)
if (db && stepResults.length > 0) {
  for (const result of stepResults) {
    if (result.screenshotBase64) {
      await db.update(testRunSteps)
        .set({ screenshotBase64: result.screenshotBase64 })
        .where(/* match by stepId + resultId */);
    }
  }
}

// THEN strip screenshots from Temporal return payload
const lightResults = stepResults.map(({ screenshotBase64, ...rest }) => rest);
```

**Critical design decision:** The `executeStepsActivity` currently runs BEFORE `persistResults`. The `test_run_steps` rows don't exist yet when `executeStepsActivity` finishes. There are two approaches:

**Approach A -- Two-pass persist (recommended):**
1. `executeStepsActivity` returns step results (without screenshots) plus a separate `screenshots` map keyed by stepId
2. `persistResults` activity creates `test_run_steps` rows (without screenshots), then updates them with screenshots from a separate parameter
3. Problem: the screenshots map still goes through Temporal's payload

**Approach B -- Direct DB write from execute activity (recommended):**
1. Inject DB into `executeStepsActivity` using the factory DI pattern (same as `createPersistActivities`)
2. `executeStepsActivity` creates a preliminary `test_run_results` row and `test_run_steps` rows with screenshots
3. `persistResults` activity does an upsert or skips rows that already exist
4. Problem: splits persist responsibility across two activities

**Approach C -- Persist screenshots via Redis (simplest):**
1. After each step completes, publish screenshot to Redis with a dedicated channel
2. A separate consumer (or the persist activity) reads from Redis and writes to DB
3. Problem: adds complexity with no clear benefit

**Approach D -- Blob storage in execute activity (cleanest, recommended):**
1. Convert `executeStepsActivity` to a factory function like `createExecuteActivities(db)`
2. After all steps execute, write screenshot data directly into `test_run_steps` rows
3. Since `test_run_steps` rows don't exist yet at this point, create them in execute activity and have `persistResults` skip step insertion if rows already exist (idempotent)
4. OR: have execute activity store screenshots in a temporary `screenshot_cache` table, and `persistResults` copies from cache to final rows

**Final Recommendation -- Approach D variant with cache table:**
The cleanest approach is to NOT split the persist logic. Instead:
1. Add a `screenshot_cache` table: `{ id, test_run_id, step_id, step_order, viewport, screenshot_base64 }`
2. `executeStepsActivity` writes screenshots to cache table (via injected DB) after each step
3. `persistResults` reads from cache table when creating `test_run_steps` rows, then deletes cache entries
4. This keeps persist logic centralized and avoids Temporal payload limits

**Even simpler -- just make executeStepsActivity a factory:**
Actually, the simplest approach is:
1. Convert `executeStepsActivity` to `createExecuteActivities(db)` factory pattern
2. After all steps complete, bulk-insert screenshots into a simple `step_screenshots` table keyed by `(testRunId, viewport, stepOrder, screenshotBase64)`
3. `persistResults` joins against `step_screenshots` when creating `test_run_steps` rows
4. Delete `step_screenshots` entries after persist

### Pattern 3: Canvas Double-Buffering for Smooth Streaming
**What:** Use an HTML5 canvas element instead of an `<img>` tag for displaying screencast frames. Load the frame into an `Image` object off-screen, then draw to canvas. This avoids the reflow/repaint cycle that causes flicker with `<img>` src swaps.
**When to use:** When rapidly updating an image source in a browser (video-like frame rates).
**Example:**
```typescript
// Source: packages/web/src/components/live-viewer.tsx (modified)
const canvasRef = useRef<HTMLCanvasElement>(null);
const imgRef = useRef<HTMLImageElement | null>(null);

useEffect(() => {
  if (!frame || !canvasRef.current) return;
  const canvas = canvasRef.current;
  const ctx = canvas.getContext('2d');
  if (!ctx) return;

  // Create image off-screen, draw when loaded (no flicker)
  const img = new Image();
  img.onload = () => {
    canvas.width = img.width;
    canvas.height = img.height;
    ctx.drawImage(img, 0, 0);
  };
  img.src = `data:image/jpeg;base64,${frame}`;
}, [frame]);

// In JSX:
<canvas ref={canvasRef} className="w-full h-full object-contain" />
```

### Anti-Patterns to Avoid
- **Passing screenshots through Temporal workflow returns:** Temporal has a 2MB gRPC payload limit per message and 50MB per workflow history. With 10 steps x 3 viewports, screenshots easily exceed this. Always persist large blobs via a side-channel (direct DB write).
- **Splitting persist responsibility without idempotency:** If both `executeStepsActivity` and `persistResults` can create `test_run_steps` rows, one must check for existence first. Use the cache table pattern instead.
- **Using `<img>` src swaps for rapid frame updates:** This triggers browser reflow on every frame, causing visible flicker. Use canvas.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Screenshot blob storage | Custom file system storage | PostgreSQL text column (existing) | Already works, DB transactions ensure consistency, simple to query. Scale to S3 later if needed. |
| Frame smoothing algorithm | Custom interpolation | Canvas `drawImage()` with offscreen `Image` load | Browser-native, hardware-accelerated, zero dependencies |
| Step metadata enrichment | Custom mapping layer | Extend existing `StepResult` type | Data is already available in `executeStep()`, just needs to be included in the return |
| Migration for new columns | Manual SQL | `drizzle-kit push` (dev) or `drizzle-kit generate` + `migrate` (prod) | Project already uses this pattern for all schema changes |

**Key insight:** This phase is a data pipeline enrichment task, not a new feature. Every piece of data already exists somewhere in the system -- it just needs to flow through to where it's displayed.

## Common Pitfalls

### Pitfall 1: Temporal Payload Size Overflow
**What goes wrong:** Including screenshot base64 data in activity return values causes Temporal gRPC messages to exceed the 2MB per-blob or 4MB per-message limit, terminating the workflow non-recoverably.
**Why it happens:** Each PNG screenshot is ~50-200KB base64. With 10 steps, that's 0.5-2MB for one viewport. Three viewports can easily exceed 4MB total in workflow history.
**How to avoid:** Persist screenshots to database directly from the activity (via injected DB client) BEFORE returning results through Temporal. Only return lightweight step metadata through Temporal.
**Warning signs:** Workflow execution terminates with "blob size limit exceeded" or gRPC message size errors.

### Pitfall 2: Race Condition Between Screenshot Persist and Result Persist
**What goes wrong:** If `executeStepsActivity` tries to write screenshots to `test_run_steps` rows that don't exist yet (they're created by `persistResults`), the writes fail with foreign key violations or missing rows.
**Why it happens:** The workflow pipeline is: executeSteps -> persistResults. Screenshots are available after executeSteps but test_run_steps rows are created by persistResults.
**How to avoid:** Use a `step_screenshots` staging table that executeStepsActivity writes to. persistResults reads from it when creating final rows. OR make the execute activity create the rows (with upsert logic in persistResults).
**Warning signs:** Screenshots exist in staging but not in final rows, or foreign key constraint violations.

### Pitfall 3: StepEvent Type Drift Between Worker and Frontend
**What goes wrong:** The `StepEvent` type is duplicated in `packages/worker/src/streaming/types.ts` and `packages/web/src/hooks/use-live-stream.ts` (by design -- see decision 06-02). When adding `action` and `description` fields, forgetting to update both copies causes silent data loss.
**Why it happens:** Decision 06-02 explicitly chose type duplication to avoid cross-package client/server dependency. Both copies must be kept in sync manually.
**How to avoid:** Update BOTH type definitions simultaneously. Add a code comment in each file referencing the other location. Verify by checking that the live viewer displays action badges.
**Warning signs:** Live viewer shows step status but no action badges.

### Pitfall 4: drizzle-kit Schema File Registration
**What goes wrong:** New schema files or modifications to existing schema are not picked up by drizzle-kit because the schema array in `drizzle.config.ts` lists files explicitly.
**Why it happens:** `drizzle.config.ts` uses explicit file paths (`schema: ["./src/schema/users.ts", "./src/schema/test-runs.ts"]`) because drizzle-kit's CJS loader can't resolve NodeNext barrel re-exports.
**How to avoid:** This phase only adds columns to the existing `test-runs.ts` schema file (and possibly a new `step-screenshots.ts` file). If adding a new file, add it to the drizzle.config.ts schema array.
**Warning signs:** `drizzle-kit push` or `drizzle-kit generate` doesn't detect the new columns/table.

### Pitfall 5: Canvas Sizing and Aspect Ratio
**What goes wrong:** Canvas element doesn't maintain aspect ratio or appears stretched/squished compared to the `<img>` element it replaces.
**Why it happens:** Canvas `width`/`height` attributes set the drawing surface size (in pixels), while CSS width/height sets the display size. Mismatch causes distortion.
**How to avoid:** Set canvas width/height to match the image dimensions on each frame load, and use CSS `object-fit: contain` behavior via `aspect-ratio` CSS or container constraints.
**Warning signs:** Streamed browser view appears distorted or changes size erratically.

### Pitfall 6: executeStepsActivity Factory Conversion Breaks Workflow Registration
**What goes wrong:** Converting `executeStepsActivity` from a standalone exported function to a factory function (like `createPersistActivities`) changes how it's imported in `worker.ts` and how the workflow proxies it.
**Why it happens:** The workflow file uses `proxyActivities<typeof activities>` which expects a module shape. Changing to a factory pattern changes the module exports.
**How to avoid:** Follow the exact same pattern used by `createPersistActivities`: export a factory function, create a type alias `ExecuteActivities = ReturnType<typeof createExecuteActivities>`, use that type in the workflow's `proxyActivities`. Update `worker.ts` to spread the factory result.
**Warning signs:** TypeScript errors in workflow file, or "activity not registered" errors at runtime.

### 6. Factory Pattern for Execute Activity with DB Injection
```typescript
// Source: packages/worker/src/activities/execute-steps.activity.ts (modified)
import type { Database } from '@validater/db';
import { stepScreenshots } from '@validater/db';
import { nanoid } from 'nanoid';

export function createExecuteActivities(db: Database) {
  async function executeStepsActivity(params: ExecuteStepsParams): Promise<ExecutionResult> {
    // ... existing logic ...

    const stepResults = await executeSteps(page, params.steps, {
      ...params.config,
      onStepComplete: async (result) => {
        heartbeat(`step ${result.stepOrder} complete`);

        // Persist screenshot to staging table (side-channel)
        if (result.screenshotBase64) {
          try {
            await db.insert(stepScreenshots).values({
              id: nanoid(),
              testRunId: params.streamingConfig?.testRunId ?? '',
              viewport: params.viewport.name,
              stepOrder: result.stepOrder,
              screenshotBase64: result.screenshotBase64,
            });
          } catch {
            // Screenshot persist failure must not break execution
          }
        }

        // Publish step event with action/description for live streaming
        if (params.streamingConfig?.enabled) {
          publishStepEvent(params.streamingConfig!.testRunId, {
            stepId: result.stepId,
            stepOrder: result.stepOrder,
            status: result.status,
            action: result.action,
            description: result.description,
            durationMs: result.durationMs,
            error: result.error?.message,
          }).catch(() => {});
        }
      },
    });

    // Strip screenshots from Temporal return (they're already persisted)
    const lightResults = stepResults.map(({ screenshotBase64, ...rest }) => rest);

    return { /* ... */ };
  }

  return { executeStepsActivity };
}

export type ExecuteActivities = ReturnType<typeof createExecuteActivities>;
```

### 9. Canvas-Based Frame Display
```typescript
// Source: packages/web/src/components/live-viewer.tsx (modified)
import { useEffect, useRef } from "react";

function BrowserCanvas({ frame }: { frame: string | null }) {
  const canvasRef = useRef<HTMLCanvasElement>(null);

  useEffect(() => {
    if (!frame || !canvasRef.current) return;
    const canvas = canvasRef.current;
    const ctx = canvas.getContext('2d');
    if (!ctx) return;

    const img = new Image();
    img.onload = () => {
      // Match canvas drawing surface to image dimensions
      if (canvas.width !== img.width || canvas.height !== img.height) {
        canvas.width = img.width;
        canvas.height = img.height;
      }
      ctx.drawImage(img, 0, 0);
    };
    img.src = `data:image/jpeg;base64,${frame}`;
  }, [frame]);

  return (
    <div className="aspect-video overflow-hidden rounded-lg border bg-muted/30">
      <canvas
        ref={canvasRef}
        className="w-full h-full"
        style={{ objectFit: 'contain' }}
      />
    </div>
  );
}
```

## Open Questions

1. **Screenshot storage scaling**
   - What we know: PostgreSQL text columns with TOAST handle large values, but storing many screenshots (10 steps x 3 viewports = 30 per test run) in base64 adds ~33% overhead vs raw binary.
   - What's unclear: At what scale (number of test runs) does DB-based screenshot storage become a performance bottleneck?
   - Recommendation: Use PostgreSQL text columns for now (already in schema). Add a TODO comment for migrating to S3/blob storage when the project reaches production scale. This is explicitly out of scope for Phase 6.1.

2. **Screenshot persistence during retries**
   - What we know: Temporal retries activities, and activities with side effects need idempotency. If `executeStepsActivity` is retried, it will try to insert duplicate screenshots into the staging table.
   - What's unclear: Whether a simple `ON CONFLICT DO UPDATE` is sufficient or if more complex idempotency is needed.
   - Recommendation: Use a unique constraint on `(test_run_id, viewport, step_order)` in the staging table and use `onConflictDoUpdate` to overwrite. Retried activities will simply overwrite previous screenshot data.

3. **Streaming bandwidth with quality 80 and everyNthFrame 1**
   - What we know: Quality 80 JPEG at 1280x720 is ~50-100KB per frame. At everyNthFrame 1 and typical page rendering rates, this could be 5-15 frames/second = 250KB-1.5MB/second through Redis.
   - What's unclear: Whether Redis pub/sub and WebSocket can sustain this throughput without backpressure.
   - Recommendation: Start with quality 80 and everyNthFrame 1. If bandwidth issues arise, reduce to everyNthFrame 2 as fallback. Monitor Redis memory and WS message queue.
