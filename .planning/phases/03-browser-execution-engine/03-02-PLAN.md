<objective>
Build the browser pool infrastructure and Temporal activities that wire the core execution engine (from Plan 01) to pooled browser instances, enabling both single-viewport and multi-viewport test execution with lifecycle management and memory monitoring.

Purpose: This transforms the execution engine from a library function into a production-ready Temporal activity system. Browser pooling eliminates the ~2s launch overhead per activity call and prevents memory leaks through max-lifetime/max-pages retirement. Multi-viewport fan-out enables running the same test across desktop, tablet, and mobile in parallel.

Output: Two Temporal activities -- `executeStepsActivity` (single viewport) and `executeViewportsActivity` (multi-viewport) -- backed by a `generic-pool` managed browser pool with memory monitoring.
</objective>

  <action>
    Install the pooling library and create browser pool infrastructure in `packages/worker/src/browser/`:

    **0. Install generic-pool:**
    ```bash
    cd packages/worker && pnpm add generic-pool && pnpm add -D @types/generic-pool
    ```

    **1. `browser/memory-monitor.ts`** -- Memory health checking:
    - Define `MemoryConfig` interface: `{ maxRssBytes?: number; maxHeapPercentage?: number }` with defaults 1.5GB and 0.9.
    - Define `MemoryHealth` interface: `{ healthy: boolean; rssBytes: number; heapUsedBytes: number; heapTotalBytes: number; heapPercentage: number }`.
    - Create `function checkMemoryHealth(config?: MemoryConfig): MemoryHealth`:
      - Uses `process.memoryUsage()` to get `rss`, `heapUsed`, `heapTotal`.
      - Computes `heapPercentage = heapUsed / heapTotal`.
      - `healthy = rss < maxRssBytes && heapPercentage < maxHeapPercentage`.
      - Returns the full `MemoryHealth` object.

    **2. `browser/pool.ts`** -- Browser pool wrapping generic-pool:
    - Define `PooledBrowser` interface: `{ browser: Browser; createdAt: number; pagesProcessed: number }`.
    - Define `BrowserPoolConfig` interface:
      ```typescript
      {
        maxBrowsers?: number;       // default 3 (from env BROWSER_POOL_MAX)
        minBrowsers?: number;       // default 1 (from env BROWSER_POOL_MIN)
        maxLifetimeMs?: number;     // default 5 * 60 * 1000 (5 min)
        maxPagesPerBrowser?: number;// default 50
        acquireTimeoutMs?: number;  // default 30_000
        idleTimeoutMs?: number;     // default 60_000
        evictionIntervalMs?: number;// default 30_000
      }
      ```
    - Create `function createBrowserPool(config?: BrowserPoolConfig): Pool<PooledBrowser>`:
      - Read defaults from env vars: `BROWSER_POOL_MAX`, `BROWSER_POOL_MIN` (parseInt with fallback).
      - Create factory object with:
        - `create`: `chromium.launch({ headless: true })` -> wrap in PooledBrowser with `createdAt: Date.now(), pagesProcessed: 0`.
        - `destroy`: `pooled.browser.close()`.
        - `validate`: Check `browser.isConnected()`, `Date.now() - createdAt < maxLifetimeMs`, `pagesProcessed < maxPagesPerBrowser`, AND `checkMemoryHealth().healthy`. Return false if any check fails (triggers pool to destroy and recreate).
      - Call `createPool(factory, { max, min, testOnBorrow: true, acquireTimeoutMillis, idleTimeoutMillis, evictionRunIntervalMillis })`.
      - Return the pool.
    - Create `defaultBrowserPool` as a lazy singleton: `let _pool: Pool<PooledBrowser> | null = null;` with `export function getDefaultPool(): Pool<PooledBrowser>` that creates on first call.
    - Create `export async function shutdownPool(): Promise<void>` that drains the pool if it exists (`pool.drain()` then `pool.clear()`), sets `_pool = null`.
    - Import `chromium` and `Browser` from playwright. Import `createPool` and `Pool` from `generic-pool`. Import `checkMemoryHealth` from `./memory-monitor.js`.

    **3. `browser/index.ts`** -- Barrel export:
    ```typescript
    export { createBrowserPool, getDefaultPool, shutdownPool } from './pool.js';
    export type { PooledBrowser, BrowserPoolConfig } from './pool.js';
    export { checkMemoryHealth } from './memory-monitor.js';
    export type { MemoryHealth, MemoryConfig } from './memory-monitor.js';
    ```

    IMPORTANT: Use `.js` extensions in all relative imports. Import `chromium` from `playwright` (runtime dependency, already in worker package.json). The pool validate function must call `checkMemoryHealth()` to integrate memory monitoring.

  <action>
    Create two Temporal activities that wire the browser pool to the core execution engine:

    **1. `activities/execute-steps.activity.ts`** -- Single viewport execution:
    - Create `export async function executeStepsActivity(params: ExecuteStepsParams): Promise<ExecutionResult>`:
      - Define `ExecuteStepsParams`:
        ```typescript
        interface ExecuteStepsParams {
          url: string;
          steps: TestStep[];
          viewport: ViewportConfig;
          config?: ExecutionConfig;
        }
        ```
      - Acquire a browser from the pool: `const pooled = await getDefaultPool().acquire();`
      - Wrap everything in try/finally to guarantee release: `finally { pooled.pagesProcessed++; await getDefaultPool().release(pooled); }`
      - Create a new context with viewport settings:
        ```typescript
        const context = await pooled.browser.newContext({
          viewport: { width: params.viewport.width, height: params.viewport.height },
          deviceScaleFactor: params.viewport.deviceScaleFactor,
          isMobile: params.viewport.isMobile,
          hasTouch: params.viewport.hasTouch,
        });
        ```
      - Create page from context: `const page = await context.newPage();`
      - Wrap page work in try/finally to close context: `finally { await context.close(); }`
      - Call `executeSteps(page, params.steps, params.config)` from @validater/core to get `StepResult[]`.
      - Build and return `ExecutionResult`:
        ```typescript
        {
          viewport: params.viewport.name,
          url: params.url,
          stepResults: results,
          totalDurationMs: results.reduce((sum, r) => sum + r.durationMs, 0),
          startedAt: startTime, // ISO string captured before execution
          completedAt: new Date().toISOString(),
        }
        ```
    - Import `getDefaultPool` from `../browser/pool.js`. Import `executeSteps` and types from `@validater/core`.

    **2. `activities/execute-viewports.activity.ts`** -- Multi-viewport fan-out:
    - Create `export async function executeViewportsActivity(params: ExecuteViewportsParams): Promise<ExecutionResult[]>`:
      - Define `ExecuteViewportsParams`:
        ```typescript
        interface ExecuteViewportsParams {
          url: string;
          steps: TestStep[];
          viewports?: string[]; // viewport names, defaults to DEFAULT_VIEWPORTS
          config?: ExecutionConfig;
        }
        ```
      - Resolve viewport names to ViewportConfig objects using `VIEWPORT_PRESETS` from @validater/core. If a name is not found in presets, skip it with a console.warn.
      - Execute each viewport sequentially (NOT in parallel -- pool has limited browsers, sequential is safer for memory):
        ```typescript
        const results: ExecutionResult[] = [];
        for (const viewport of resolvedViewports) {
          const result = await executeStepsActivity({
            url: params.url,
            steps: params.steps,
            viewport,
            config: params.config,
          });
          results.push(result);
        }
        return results;
        ```
      - Import `VIEWPORT_PRESETS, DEFAULT_VIEWPORTS` from `@validater/core`. Import `executeStepsActivity` from `./execute-steps.activity.js`.

    IMPORTANT:
    - Activities must return ONLY plain serializable objects (Temporal requirement). No Playwright objects, no Buffer objects. Screenshots are already base64 strings from the core executor.
    - Each viewport gets its own BrowserContext (full isolation -- cookies, storage, viewport) via `browser.newContext()`. Never resize a single page.
    - Use `.js` extensions in all relative imports.
    - Do NOT modify existing activities (crawl-dom, validate-steps). They will be migrated to use the pool in Phase 4.
  </action>
