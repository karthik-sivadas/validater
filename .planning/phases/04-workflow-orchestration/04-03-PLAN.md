Rewrite `packages/worker/src/worker.ts` to create a production worker that registers all activities and points to the test-run workflow.

```typescript
import { createRequire } from 'node:module';
import { Worker } from '@temporalio/worker';
import { db } from '@validater/db';
import { shutdownPool } from './browser/pool.js';

// Runtime activity imports (NOT workflow sandbox -- these run in Node.js)
import * as crawlActivities from './activities/crawl-dom.activity.js';
import * as generateActivities from './activities/generate-steps.activity.js';
import * as validateActivities from './activities/validate-steps.activity.js';
import * as executeActivities from './activities/execute-steps.activity.js';
import { createPersistActivities } from './activities/persist-results.activity.js';

const require = createRequire(import.meta.url);

async function run() {
  const persistActivities = createPersistActivities(db);

  const worker = await Worker.create({
    workflowsPath: require.resolve('./workflows/test-run.workflow'),
    activities: {
      ...crawlActivities,
      ...generateActivities,
      ...validateActivities,
      ...executeActivities,
      ...persistActivities,
    },
    taskQueue: 'test-pipeline',
  });

  console.log('Worker started on task queue: test-pipeline');

  // Graceful shutdown: drain browser pool on SIGINT/SIGTERM
  const shutdown = async () => {
    console.log('Shutting down worker...');
    await shutdownPool();
    process.exit(0);
  };
  process.on('SIGINT', shutdown);
  process.on('SIGTERM', shutdown);

  await worker.run();
}

run().catch((err) => {
  console.error('Worker failed:', err);
  process.exit(1);
});
```

Key points:
- Uses `createRequire(import.meta.url)` for ESM compatibility (existing pattern)
- Points workflowsPath to test-run.workflow (Temporal bundler will pick up viewport-execution.workflow via the import in test-run.workflow)
- Injects `db` into persistActivities factory
- Spreads all activity objects into a single `activities` map
- Task queue is 'test-pipeline' (single queue for now per research recommendation)
- SIGINT/SIGTERM handlers drain the browser pool before exit

Create `packages/web/src/server/run-test.ts` with two server functions:

**1. runTest -- starts a workflow non-blocking**

```typescript
import { createServerFn } from '@tanstack/react-start';
import { z } from 'zod';

const RunTestInputSchema = z.object({
  url: z.string().url(),
  testDescription: z.string().min(1).max(2000),
  viewports: z.array(z.string()).min(1).max(10).default(['desktop', 'tablet', 'mobile']),
});

export const runTest = createServerFn({ method: 'POST' })
  .inputValidator(RunTestInputSchema)
  .handler(async ({ data }) => {
    // Dynamic imports to avoid bundling server-only deps in client
    const { nanoid } = await import('nanoid');
    const { createTemporalClient, testRunWorkflow } = await import('@validater/worker');
    const { db, testRuns } = await import('@validater/db');
    const { VIEWPORT_PRESETS } = await import('@validater/core');

    const testRunId = nanoid();

    // Resolve viewport names to ViewportConfig objects
    const resolvedViewports = data.viewports
      .map((name) => VIEWPORT_PRESETS[name])
      .filter(Boolean);

    if (resolvedViewports.length === 0) {
      throw new Error(`No valid viewports found. Available: ${Object.keys(VIEWPORT_PRESETS).join(', ')}`);
    }

    // Insert initial test_run record
    await db.insert(testRuns).values({
      id: testRunId,
      userId: 'anonymous', // TODO: Wire auth context in Phase 5
      url: data.url,
      testDescription: data.testDescription,
      status: 'pending',
      viewports: data.viewports,
    });

    // Start workflow non-blocking
    const client = await createTemporalClient();
    const handle = await client.workflow.start(testRunWorkflow, {
      args: [{
        testRunId,
        url: data.url,
        testDescription: data.testDescription,
        viewports: resolvedViewports,
      }],
      taskQueue: 'test-pipeline',
      workflowId: testRunId,
    });

    return {
      testRunId,
      workflowId: handle.workflowId,
    };
  });
```

**2. getTestRunStatusFn -- queries workflow status**

```typescript
const GetStatusInputSchema = z.object({
  testRunId: z.string().min(1),
});

export const getTestRunStatusFn = createServerFn({ method: 'GET' })
  .inputValidator(GetStatusInputSchema)
  .handler(async ({ data }) => {
    const { createTemporalClient, getTestRunStatus } = await import('@validater/worker');

    const client = await createTemporalClient();
    const handle = client.workflow.getHandle(data.testRunId);

    try {
      const status = await handle.query(getTestRunStatus);
      return { status, found: true };
    } catch (err) {
      // Workflow may have completed or not exist
      // Fall back to database status
      const { db, testRuns } = await import('@validater/db');
      const { eq } = await import('drizzle-orm');

      const run = await db.query.testRuns.findFirst({
        where: eq(testRuns.id, data.testRunId),
      });

      if (!run) {
        return { status: null, found: false };
      }

      return {
        status: {
          phase: run.status,
          viewportsComplete: run.status === 'complete' ? (run.viewports as string[]).length : 0,
          viewportsTotal: (run.viewports as string[]).length,
          error: run.error ?? undefined,
        },
        found: true,
      };
    }
  });
```

Key design decisions:
- **Non-blocking start**: Uses `client.workflow.start()` not `.execute()`. Returns immediately with testRunId.
- **Dynamic imports**: All server-side deps (Temporal client, db, core) imported dynamically to prevent client bundling.
- **Status fallback**: If Temporal query fails (workflow completed/not found), falls back to database record.
- **Viewport resolution**: Accepts viewport names (strings), resolves to ViewportConfig objects via VIEWPORT_PRESETS.
- **Initial DB insert**: Creates the test_runs row before starting the workflow so the record exists immediately for the frontend.
- **Anonymous userId**: Phase 5 will wire the authenticated user context. For now, uses a placeholder.
- Uses `inputValidator` (not `validator`) per TanStack Start v1.166.2 API (documented decision from 02-05).

Note: nanoid is already a dependency of @validater/web via @validater/worker workspace dependency chain, but if TypeScript cannot resolve it, add nanoid as a direct dependency to the web package.json.
