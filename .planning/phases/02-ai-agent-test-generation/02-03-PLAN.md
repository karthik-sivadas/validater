    **packages/core/src/ai/prompts/system.ts** -- System prompt (MUST exceed 1024 tokens for Sonnet cache eligibility):

    Export a `SYSTEM_PROMPT` string constant. The prompt should instruct Claude to:
    1. Act as a test automation expert that generates structured test steps from natural language descriptions
    2. Use the simplified DOM and aria snapshot provided to ground every locator in real page elements
    3. Generate multiple locator strategies per element in priority order: role > text > label > placeholder > testId > css > xpath
    4. Assign confidence scores: role/text-based locators get 0.8-1.0, CSS structural selectors get 0.4-0.7, XPath gets 0.2-0.5
    5. Never hallucinate selectors -- every locator must reference an element visible in the provided DOM or aria snapshot
    6. Include reasoning for each step explaining WHY this action advances the test goal
    7. Prefer semantic locators (getByRole, getByText, getByLabel) over structural ones (CSS, XPath)
    8. For assertions, verify the expected outcome of each interaction (e.g., after clicking login, assert URL contains /dashboard)
    9. Keep step count reasonable (3-15 steps for most tests)
    10. Handle common patterns: form filling sequences, navigation flows, CRUD operations, authentication flows

    Include 2-3 examples in the prompt showing input (DOM snippet + description) and expected output (test steps with locators). This helps in-context learning AND pushes the prompt above the 1024 token cache threshold.

    The prompt should reference the exact schema structure (action types, locator types, assertion types) so Claude knows the output format constraints.

    **packages/core/src/ai/prompts/templates.ts** -- User prompt builders:

    `buildUserPrompt(simplifiedDomHtml: string, ariaSnapshot: string, testDescription: string): string`
    - Structure the user message with clear sections:
      ```
      ## Target Page Structure

      ### Simplified DOM
      {simplifiedDomHtml}

      ### Accessibility Tree (ARIA Snapshot)
      {ariaSnapshot}

      ## Test Description
      {testDescription}

      ## Instructions
      Generate test steps that accomplish the described test. Ground every locator in the DOM and ARIA snapshot above. Prefer semantic locators.
      ```

    `buildValidationPrompt(step: object, verificationResults: object): string`
    - For self-healing: given a step with failed locators and verification results, ask Claude to suggest alternative locators
    - This is used in Plan 02-04 but defined here so all prompt logic is co-located

    `createAIClient(options?: { model?: string })`:
    - Import `createAnthropic` from `@ai-sdk/anthropic`
    - Create Anthropic provider instance (reads ANTHROPIC_API_KEY from environment automatically)
    - Return configured model reference
    - Default model: `process.env.AI_MODEL || 'claude-sonnet-4-5'`

    `generateTestSteps(params: { simplifiedDomHtml: string, ariaSnapshot: string, testDescription: string, model?: string }): Promise<{ steps: Array<RawTestStep>, reasoning: string, usage: TokenUsage }>`
    - Import `generateObject` and `Output` from `ai`
    - Import `RawTestStep` from types (NOT TestStep -- the AI does not generate IDs)
    - Use `TestGenerationSchema` from `@validater/core/schemas`
    - Call `generateObject` with:
      - `model`: anthropic provider with configured model
      - `output`: `Output.object({ schema: TestGenerationSchema })`
      - `system`: `SYSTEM_PROMPT` (cached via providerOptions)
      - `messages`: user message built with `buildUserPrompt`
      - `providerOptions.anthropic.cacheControl`: `{ type: 'ephemeral' }` for prompt caching
    - Extract token usage from the response `usage` object including `cacheCreationInputTokens` and `cacheReadInputTokens`
    - Return the structured result with usage data
    - The return type uses `RawTestStep` (steps without IDs) -- IDs are assigned downstream by the pipeline

    Use `.js` extensions on all relative imports. Do NOT hardcode the API key -- it comes from environment.

    **packages/core/src/ai/rate-limiter.ts** -- Rate-limited API queue:

    `createApiQueue(options?: { concurrency?: number, intervalMs?: number, intervalCap?: number }): PQueue`
    - Import PQueue from `p-queue`
    - Default options: `{ concurrency: 5, interval: 60_000, intervalCap: 40 }`
    - These defaults are conservative for Tier 1 (50 RPM). Can be tuned up for higher tiers.
    - Return configured PQueue instance

    `queuedRequest<T>(queue: PQueue, fn: () => Promise<T>): Promise<T>`
    - Wrapper that adds a function to the queue
    - Sets `throwOnTimeout: true` to surface errors
    - Returns the result

    Export a default singleton queue: `export const defaultApiQueue = createApiQueue()`
    - Singleton ensures all API calls across the application share the same rate limit
    - Can be overridden by creating a custom queue via `createApiQueue`

    **packages/core/src/ai/cost-tracker.ts** -- Token usage and cost tracking:

    Define model pricing as a configuration object:
    ```typescript
    const MODEL_PRICING: Record<string, { input: number; output: number; cacheWrite: number; cacheRead: number }> = {
      'claude-sonnet-4-5': { input: 3.0, output: 15.0, cacheWrite: 3.75, cacheRead: 0.30 },
      'claude-haiku-3-5': { input: 1.0, output: 5.0, cacheWrite: 1.25, cacheRead: 0.10 },
    };
    ```
    Prices are per million tokens.

    `calculateCost(usage: TokenUsage, model?: string): CostEstimate`
    - Look up pricing for the model (default to sonnet pricing)
    - Calculate: inputCost, outputCost, cacheWriteCost, cacheReadCost, totalCost
    - Return `CostEstimate` object with all fields + currency: 'USD'

    `class CostTracker`:
    - Maintains an array of `{ timestamp: Date, usage: TokenUsage, cost: CostEstimate, model: string }` entries
    - `track(usage: TokenUsage, model: string): CostEstimate` -- records entry, returns cost
    - `getTotalCost(): number` -- sum of all totalCost values
    - `getUsageSummary(): { totalInputTokens, totalOutputTokens, totalCacheReads, totalCacheWrites, totalCost, requestCount }`
    - `reset(): void` -- clears all entries

    This is an in-memory tracker. Persistent cost tracking comes later (database storage in Phase 4/5).
