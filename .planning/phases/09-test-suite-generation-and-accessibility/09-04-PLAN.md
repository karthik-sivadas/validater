**Create `packages/web/src/server/test-suites.ts`:**

Follow the exact patterns from `run-test.ts` and `test-runs.ts`:

1. **`generateSuite`** (POST): Creates a test_suites DB record and starts the testSuiteWorkflow.
   - Input: `{ url: z.string().url(), featureDescription: z.string().min(1).max(5000) }`
   - Auth: get session, extract userId
   - Dynamic imports: `nanoid`, `@validater/worker` (createTemporalClient, testSuiteWorkflow), `@validater/db` (db, testSuites)
   - Insert suite record with status 'pending', userId, url, featureDescription
   - Start `testSuiteWorkflow` via Temporal client with args `[{ suiteId, url, featureDescription }]`, taskQueue `'test-pipeline'`, workflowId `suiteId`
   - Return `{ suiteId }`

2. **`getSuiteList`** (GET): Paginated, user-scoped suite list.
   - Input: `{ page: z.number().default(1), pageSize: z.number().default(10) }`
   - Query test_suites where userId = session user, ordered by createdAt desc
   - Return `{ suites, total, page, pageSize, totalPages }`

3. **`getSuiteDetail`** (GET): Single suite with all test cases.
   - Input: `{ suiteId: z.string().min(1) }`
   - Query test_suites by id, check ownership (return null if not owner)
   - Query test_cases where suiteId matches, ordered by `order` asc
   - Return `{ suite, testCases }`

4. **`getSuiteStatusFn`** (GET): Query workflow status for progress polling.
   - Input: `{ suiteId: z.string().min(1) }`
   - Try Temporal query (getSuiteStatus), fall back to DB status
   - Return `{ status, found }` (same pattern as getTestRunStatusFn)

5. **`runTestCase`** (POST): Run a single test case from a suite.
   - Input: `{ testCaseId: z.string().min(1), viewports: z.array(z.string()).default(['desktop', 'tablet', 'mobile']) }`
   - Query test_cases by id, get suiteId, verify suite ownership
   - Get the test case's description and the suite's url
   - Use existing `triggerTestRun` from `run-test-core.ts`:
     ```typescript
     const { triggerTestRun } = await import("@/server/run-test-core");
     const result = await triggerTestRun({
       userId: session.user.id,
       url: suite.url,
       testDescription: testCase.description,
       viewports: data.viewports,
     });
     ```
   - Update test_cases.testRunId = result.testRunId
   - Return `{ testRunId: result.testRunId }`

**Create `packages/web/src/components/accessibility-panel.tsx`:**

A reusable component for displaying accessibility results:

- Props: `{ data: { violationCount, passCount, incompleteCount, inapplicableCount, violations: Array<{ id, impact, description, help, helpUrl, tags, nodes, nodeCount }> } | null }`
- If data is null, show nothing (no a11y data available)
- Summary section: badges for critical/serious/moderate/minor violation counts using color coding:
  - critical: red, serious: orange, moderate: yellow, minor: blue
- Also show passCount, incompleteCount as muted text
- Violation list: each violation shows impact badge, help text, description, "Learn more" link (helpUrl opens in new tab), and affected elements count (nodeCount)
- Use existing shadcn components: Card, CardHeader, CardTitle, CardContent, Badge
- Collapsible violation details (initially collapsed if more than 5 violations)
- Use Tailwind for styling, match existing design system
