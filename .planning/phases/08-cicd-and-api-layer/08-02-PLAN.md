  <action>
    1. Create `packages/web/src/server/run-test-core.ts`:
       - Extract the core business logic from `run-test.ts` handler into a shared function:
         ```
         export async function triggerTestRun(params: {
           userId: string;
           url: string;
           testDescription: string;
           viewports: string[];
         }): Promise<{ testRunId: string; workflowId: string }>
         ```
       - This function does everything the current handler does EXCEPT auth checking:
         a. Generate nanoid
         b. Resolve viewport names to ViewportConfig objects (validate they're valid)
         c. Insert test_runs record
         d. Start Temporal workflow
         e. Return { testRunId, workflowId }
       - Use dynamic imports for server-only deps (same pattern as existing code)
       - Throw descriptive errors for invalid viewports

    2. Refactor `packages/web/src/server/run-test.ts`:
       - Keep the existing `runTest` createServerFn and `getTestRunStatusFn`
       - Replace the inline business logic in `runTest` handler with a call to `triggerTestRun` from run-test-core.ts
       - Auth check (session-based) stays in the server function handler
       - This ensures existing frontend still works identically

    3. Create `packages/web/src/routes/api/v1/runs.ts`:
       - Use `createFileRoute('/api/v1/runs')` with `server.handlers` (same pattern as `/api/auth/$`)
       - POST handler:
         a. Call `verifyApiKey(request)` from `@/lib/api-auth` (dynamic import)
         b. If not valid, return `Response.json({ error }, { status: 401 })`
         c. Parse request body as JSON
         d. Validate with Zod schema (url: string URL, testDescription: string min 1 max 2000, viewports: string[] optional default ["desktop","tablet","mobile"])
         e. If validation fails, return `Response.json({ error: 'Invalid request body', details }, { status: 400 })`
         f. Call `triggerTestRun({ userId: keyResult.userId, ...validatedData })` from run-test-core.ts (dynamic import)
         g. Return `Response.json({ testRunId, status: 'pending' }, { status: 201 })`
         h. Wrap in try/catch, return `Response.json({ error: 'Internal server error' }, { status: 500 })` on unexpected errors

    IMPORTANT: The verifyApiKey helper from 08-01 returns the userId associated with the API key. Pass this userId to triggerTestRun so test runs are correctly attributed to the key owner.

    IMPORTANT: Use `await import()` for all imports inside the handler (same pattern as existing server functions) to avoid bundling server-only deps.

    GOTCHA: TanStack Start server routes use `createFileRoute` from `@tanstack/react-router`, NOT from `@tanstack/react-start`. The existing `/api/auth/$.ts` is the reference pattern.

  <action>
    1. Create `packages/web/src/routes/api/v1/runs/$runId.ts`:
       - Use `createFileRoute('/api/v1/runs/$runId')` with `server.handlers`
       - GET handler:
         a. Call `verifyApiKey(request)` from `@/lib/api-auth` (dynamic import)
         b. If not valid, return `Response.json({ error }, { status: 401 })`
         c. Extract `runId` from `params` (TanStack Start server route params)
         d. Query test run from DB using the same pattern as `getTestRunDetail` in test-runs.ts:
            - Select test run by id
            - Verify ownership: `run.userId === keyResult.userId` (API key owner must own the test run)
            - If not found or not owned, return `Response.json({ error: 'Not found' }, { status: 404 })`
         e. First try Temporal query for live status (same pattern as `getTestRunStatusFn`):
            - Get workflow handle, query `getTestRunStatus`
            - If workflow not found (catch), fall back to DB status
         f. If run status is "complete", include full results with inline screenshotBase64:
            - Query test_run_results and test_run_steps (same pattern as getTestRunDetail)
            - Structure response as:
              ```json
              {
                "testRunId": "...",
                "url": "...",
                "testDescription": "...",
                "status": "complete",
                "phase": "complete",
                "viewportsComplete": N,
                "viewportsTotal": N,
                "createdAt": "ISO timestamp",
                "completedAt": "ISO timestamp",
                "results": [
                  {
                    "viewport": "desktop",
                    "totalDurationMs": 1234,
                    "steps": [
                      {
                        "stepOrder": 1,
                        "action": "navigate",
                        "description": "Go to homepage",
                        "status": "pass",
                        "durationMs": 500,
                        "errorMessage": null,
                        "screenshotBase64": "data:image/jpeg;base64,..."
                      }
                    ]
                  }
                ]
              }
              ```
            - Screenshots are included inline as base64 strings. The base64 data is already stored
              in the database (test_run_steps.screenshot_base64 column). CI/CD consumers (like the
              GitHub Action in 08-03) need the complete picture in a single API request -- no
              separate screenshot endpoint needed.
            - If a step has no screenshot, set `screenshotBase64` to null.
         g. For non-complete statuses, return status/phase only (no results array):
            ```json
            {
              "testRunId": "...",
              "status": "executing",
              "phase": "executing",
              "viewportsComplete": 1,
              "viewportsTotal": 3
            }
            ```

    GOTCHA: Use dynamic imports for all server-only dependencies inside the handler function. Match the pattern from existing server functions (run-test.ts, test-runs.ts).

    GOTCHA: TanStack Start server route params are accessed via the `params` argument to the handler function, not via a separate context.
