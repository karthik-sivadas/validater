  <action>
    1. Create `.github/actions/validater-test/action.yml` as a composite action:

    Inputs:
      - `api-url` (required): Validater API base URL (e.g., https://validater.example.com)
      - `api-key` (required): Validater API key (should be stored as a GitHub secret)
      - `url` (required): URL to test
      - `description` (required): Test description in natural language
      - `viewports` (optional, default "desktop,tablet,mobile"): Comma-separated viewport list
      - `timeout` (optional, default "300"): Timeout in seconds

    Outputs:
      - `test-run-id`: The Validater test run ID
      - `status`: Final status (complete/failed/timeout)
      - `results-url`: URL to view results in Validater UI
      - `results-json`: Full JSON results payload (for downstream steps)

    Steps:
      a. **Trigger test run** (id: trigger):
         - POST to `${{ inputs.api-url }}/api/v1/runs` with Authorization Bearer header
         - Body: `{"url":"...","testDescription":"...","viewports":[...]}`
         - Convert comma-separated viewports input to JSON array using `jq -R 'split(",")'`
         - Parse response: extract testRunId
         - Set outputs: test-run-id, results-url
         - On non-201 response: `echo "::error::Failed to trigger test run: $BODY"` and exit 1
         - Use `curl -s` (silent) to avoid logging headers/key

      b. **Poll for completion** (id: poll):
         - Loop: GET `${{ inputs.api-url }}/api/v1/runs/$TEST_RUN_ID` with Bearer auth
         - Check `status` field in response
         - If "complete" or "failed": break loop, set status output
         - Log progress: `echo "Test run phase: ${PHASE} (${ELAPSED}s elapsed)"`
         - Sleep 10 seconds between polls
         - On timeout: set status=timeout, `echo "::error::..."`, exit 1
         - On failed: `echo "::error::Test run failed"`, exit 1
         - On complete: set results-json output

      c. **Report results** (id: report):
         - Write a summary to `$GITHUB_STEP_SUMMARY` with:
           - Test run ID and link to results
           - Overall pass/fail status
           - Per-viewport summary (viewport name, step count, pass/fail)
         - Use markdown formatting for the summary
         - This makes results visible directly in the GitHub Actions UI

    IMPORTANT: Never echo or log the API key. Use `curl -s` (silent mode). The `api-key` input
    should be passed via `${{ secrets.VALIDATER_API_KEY }}` in consumer workflows.

    IMPORTANT: Use `shell: bash` for all steps (composite actions require explicit shell).

    Reference the research doc Pattern 4 for the complete design.
  </action>

      - `description`: Test description (required, default "Verify the deployed application works correctly")

    Jobs:
      a. **validater-test** job:
         - runs-on: ubuntu-latest
         - steps:
           1. Checkout (actions/checkout@v4) -- needed for local action reference

              ```yaml
              uses: ./.github/actions/validater-test

                api-url: ${{ secrets.VALIDATER_API_URL }}

                url: ${{ github.event.inputs.url || github.event.deployment_status.target_url }}
                description: ${{ github.event.inputs.description || 'Verify deployment works correctly' }}
                viewports: 'desktop,mobile'
                timeout: '300'
              ```
           3. Comment on PR with results (if PR context available):
              ```yaml

              with:

                  // Find associated PR and post results comment
              ```
